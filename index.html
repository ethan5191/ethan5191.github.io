Why I do this...

Personal MROWS accomplishments (June 2015 - Present)

Reserve Travel Module
Working with another dev team we created a Soap web service between MROWS and DTMS. This web service allowed Marines to complete and submit a travel claim electronically after finishing their orders. This resulted in the Marine getting paid in hours instead of weeks. I worked closely with my team lead to design the database structure for the travel claims. We also created the Soap web service call to DTMS, and created two web services in MROWS that allowed DTMS to send us either reject or recon claim information.
DAI Initiative
The DAI initiative was the USMC wide transition from the SABRS accounting system to the DAI accounting system. This involved determining which orders in MROWS needed to go to DAI, populating the newly created DAI tables, and building the process to create the XML file that was sent to DAI. The team also created a process to receive and parse DAI acknowledgement files so that we could track the status of the orders we had sent to DAI.
HP Fortify
Our team was one of the first teams to get HP fortify setup. As a result, we lead the charge in creating common solutions for the rest of our division to use to remove findings across the division. Over the span of about 3 months we worked to remove almost 5000 security findings that existed in MROWS and other applications within the division.
Hudson to Jenkins
I worked closely with an experienced engineer to get the MROWS build process moved over to Jenkins. This involved creating a JenkinsFile and setting up the different jobs for test and production builds. The result was a faster build process that is now 100% auditable. The Jenkins process does the entire production build process. It merges the release branch into trunk, creates a tag, does the fortify scan, and creates and stages the MROWS war to that is to be deployed.
Spring Migration
In 2018 the team began the long process of learning to use Spring MVC and migrating MROWS from Struts 1 to Spring 5 using the MVC architecture. I took the lead on this project and created the blueprint for converting two of the bigger areas of the application. This process involved converting our old action classes to Spring controllers and services. We also had to remove the old struts html tags and replace them with JSTL tags.
iReport Migration
When the DoD purchased iText 7 licenses for all applications, we were forced to add it to our application. That way any new PDFs that we create will be created using the newest version of iText. This caused us to have to migrate all of our old reports off of iReport and recreate them using Jasper Soft Studio. I took the lead on this project to transfer all of our reports. To date I have about 20 of the 107 reports moved over.

Marine Resource Order Writing Service (MROWS)

  • MROWS is a monolithic web application. It was originally built as a windows application, and in 2007 was moved to a web application. It has a Java back-end, JSP front end using a combination of HTML, CSS, and JavaScript, runs on Weblogic servers, and connects to an Oracle SQL database. The web application was built on the Struts 1.1 framework. Over the last few years the team has slowly been moving it to Spring, replacing the old struts actions with Spring controllers and service classes. As of today, the app is about 95% migrated over, with just a few pieces left on the old Struts 1 framework.
  • While MROWS is an old monolithic app, the team has done its best over the years to make it more modern and flexible. To this end, we have made numerous parts of the application database driven. Some parts like paragraph text, certain rates, and even specific logic flows can be adjusted by the application administrators. Application administrators are a group of users who have special access and can adjust certain things within the app. Other things like enabling/disabling of jobs and user role permissions have to be done via a one time database script.
  • MROWS is an order writing system. Certain users have the ‘order writing’ role. This role allows them the ability to create orders for marines. This orders, once authenticated become official documents. They tell the marine what type of duty they are doing, when and where they are doing it, who they are reporting to, how they are getting there, and an estimate of how much they will be paid. Some of the most common duty types are annual training, schools, and activation. Once an order writer creates and routes an order it is then approved through different approvals inboxes. The specific inboxes vary based on the specifics of the order. All orders go through at least the in app authentication routing. This is the final routing before an order becomes an official document. MROWS allows orders to be created on a single individual or done in a bulk process. This bulk process is used anytime a group of marines are traveling from the same location, to the same destination, using the same travel type, and performing the same duty.
  • As a Marine Corps application, the user interface is nothing special. Our stakeholders have always told us, ‘it doesn’t matter how it looks, just as long as it does what we need it to do’. In spite of that the team has recently began to transition certain parts of the application to a Jquery based UI, using the Jquery data tables framework. This process has allowed us to simplify the back-end code that builds these views and improve the look and feel of the application, giving it a more modern look. While not a high priority project from the stakeholders, it has gained some decent traction. The initial push to move select inboxes to Jquery has been well received by the end user and product owners alike. This has allowed us to put more effort into moving more of the application over to Jquery.

Marine Corps Per Diem & Travel (MCPDT)

  • MCPDT is a small application that is made up almost entirely of SOAP web services that it owns. There are two different views that are available in production. These two parts of the app are written on the Struts 1 framework. The application exists to handle funding for PCS orders sent to it from multiple different applications. As of today there are 4 different applications that call MCDPT web services to get specific funding information. We are working on adding another web service for a sister application to be able to send MCPDT order information so that we can update DAI with this information.
  • MCPDT’s funding is entirely database driven. At the end of each fiscal year (October – September), we run a set of database scripts to create the new funding lines for the new fiscal year.
  • There really isn’t anything else to say about MCPDT. It’s a small app that hosts soap web services for other applications PCS orders. Those applications send MCPDT the order information, MCPDT does validation to ensure it’s a proper order that it can fund, then sends back the correct funding information.
